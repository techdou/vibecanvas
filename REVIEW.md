# VibeCanvas 2.0 Quality Review

## Release assessment

VibeCanvas 2.0 is a substantial engineering release rather than a patch over the v1 prototype. The previously reproduced critical failures—lost concurrent Artifact writes, stale graph overwrite, false cancellation, invalid-target success, fake Agent review, incorrect OpenCode structured field, nonfunctional replacement, and nonpersistent final state—have dedicated implementations and regression tests.

The project is suitable for local single-user creative production, Web plus MCP concurrent access to one project, OpenCode-driven workflow composition, and controlled Image 2 relay use. It is not presented as a hosted multi-tenant SaaS.

## Verified architecture path

```text
Web / MCP / Agent
    → revisioned graph patch
    → immutable asynchronous run snapshot
    → Prompt Architect
    → Image 2 generate/edit
    → technical + Agent Vision Review
    → Candidate Selector pause/resume
    → persistent Artifact status and lineage
    → branch or true in-place canvas output
```

Web and MCP share SQLite/WAL, Artifact storage, Runner, queue semantics, provider profile, and graph protocol.

## Critical fixes verified

- OpenCode response reads `info.structured`; structured PromptSpec regression passes.
- Vision Review submits actual image file parts and records semantic score/cost.
- Cancellation aborts Image API, retry delays, URL downloads, and OpenCode; cancelled output is not registered.
- Invalid run targets are rejected before queueing in storage, HTTP, and MCP flows.
- `placement=replace` preserves target node position and size and writes a revisioned patch.
- Artifact final state survives refresh and is queryable from the index.
- MCP run tools return immediately with a `runId`; status/events/cancel are separate.
- Request headers, protected URL download headers, output format normalization, mask checks, URL safety, and edit input limits are tested.
- Web and MCP use one explicit shared configuration path.

## Data reliability verified

- SQLite journal mode is WAL.
- Concurrent initialization on one storage instance is serialized.
- 30 concurrent Artifact writes on one connection persist 30/30.
- 40 concurrent Artifact writes across four SQLite connections persist 40/40.
- Stale Graph Patch operations return revision conflicts without mutation.
- Run snapshots remain unchanged after later graph edits.
- queued/running lease recovery is exercised.
- Worker heartbeats update only lease fields and cannot overwrite persisted node progress.
- Stale workers cannot overwrite an externally cancelled Run.
- Artifact lineage and selected/final states persist.

## Workflow and UI coverage

- Candidate Selector pauses and resumes the same Run.
- Annotation and Mask editors create typed Artifacts.
- Artifact lineage panel exposes ancestors and descendants.
- Run panel exposes status, events, cancellation, estimated and actual cost.
- Provider settings expose capabilities, limits, headers, endpoint paths, timeout, retry, safety, and cost profile.
- Templates and `workflow.subflow` execute isolated child snapshots.
- graph revision history and restoration are available.

## Browser verification boundary

Production HTML, JS, CSS, and APIs can be built and served. In the current sandbox, automated Chromium navigation to localhost is blocked by administrator/container networking policy. That limitation is not represented as a successful visual browser test. A final desktop browser check should be performed on the target Windows or macOS machine.

## External verification boundary

A real provider probe requires the user's relay token and exact relay behavior. The automated suite uses protocol-faithful mock servers for Base64, URL, retries, multipart edits, masks, custom headers, cancellation, and OpenCode contracts. Run:

```bash
npm run doctor -- --probe-provider
```

after configuring the real relay. This call may incur cost.

## Known boundaries

- Node 22 `node:sqlite` may emit an ExperimentalWarning.
- Relay capability flags are declarations until probed against that relay.
- Cancellation cannot guarantee billing reversal after an upstream service has accepted a request.
- Public hosting still requires authentication, per-user isolation, quotas, CSRF/TLS, and secret management.
- Multi-user real-time CRDT collaboration is not included.
