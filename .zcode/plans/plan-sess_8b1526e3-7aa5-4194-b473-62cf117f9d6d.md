# VibeCanvas 通用适配 Bridge 重构方案

## 目标

把 VibeCanvas 从"OpenCode 专属耦合"重构为"通用 agent 适配"：

1. **删除 OpenCodeBridge 作为内部 LLM 依赖**，抽出可插拔 LLM 层，Prompt Architect / Vision Review 节点走 OpenAI-compatible API（豆包/Claude/GPT/ollama/OpenCode session 都能接）
2. **删除 Web 反向触发 OpenCode 的整套逻辑**（4 个 API 路由 + 按钮 + prompt 构造），agent 主控权全部走 MCP
3. **统一 CLI 入口**（`vibecanvas serve/mcp/dev/doctor/install-skills/validate`），package.json 加 bin
4. **install-skills 保留 ZCode + OpenCode + Claude Code 三种配置生成**（Codex 不做），抽出统一 emitter 抽象
5. **按 Cowart 风格重写 5 个 Skill**：每步给具体 MCP 工具入参 + 上游字段绑定 + 出错处理，补齐 7 个遗漏工具

## 范围与不做的事

- **不做** Codex 适配（豆哥没选）
- **不做** Claude Code 适配的代码改造（保留现有 --write-claude-code，但不投入测试）
- **不重构** VibeCanvas 的 typed workflow runner 业务逻辑（只换底层 LLM provider）
- **不动** SQLite/Artifact 共享机制、graph patch CAS、RunQueue lease

### 阶段 1：抽 LLM 层（先建后拆，不破坏运行）

#### 1.1 新增 `src/core/llm-provider.ts`（核心抽象）

```typescript
export interface LLMProfile {
  provider: 'openai-chat' | 'opencode-session' | 'fallback'
  baseUrl?: string
  apiKey?: string
  model?: string
  sessionId?: string        // opencode-session 专用
  username?: string         // opencode-session basic auth
  password?: string         // opencode-session basic auth
  headers?: Record<string, string>
  requestTimeoutMs?: number
  maxRetries?: number
}

export interface StructuredLLMRequest {
  system?: string
  prompt: string
  images?: Array<{ mime: string; base64: string; filename?: string }>
  schema: Record<string, unknown>       // JSON schema for structured output
  signal?: AbortSignal
}

export interface StructuredLLMResponse {
  structured: unknown
  cost?: number
  raw: unknown
}

export interface LLMProvider {
  generateStructured(req: StructuredLLMRequest): Promise<StructuredLLMResponse>
}
```

#### 1.2 新增三个实现（同文件或拆 `src/core/llm/`）

- **`OpenAIChatProvider`**：标准 `POST {baseUrl}/chat/completions`，`response_format: { type: 'json_schema', json_schema: { schema } }`，支持 images（`image_url` 数组）。豆包/智谱/GPT/Claude via openrouter/ollama 都走这条。
- **`OpenCodeSessionProvider`**：包装现有 `OpenCodeBridge.sendMessage` 逻辑（不删 bridge 文件，只改造成 LLMProvider 实现），parts 数组 + format json_schema。
- **`FallbackProvider`**：architect 调用 `buildPromptSpec`（已有），reviewer 调用 `technicalReview`（已有）。返回包装格式。

工厂函数 `createLLMProvider(profile: LLMProfile): LLMProvider`。

#### 1.3 改 `src/core/config.ts`：替换 openCode 字段为 llm 字段

```typescript
// 新增
llm: {
  architect: LLMProfile    // prompt architect 用
  reviewer: LLMProfile     // vision review 用（通常配 vision 模型）
}

// 删除
openCode: { baseUrl, sessionId, username, password, agent, model }
```

### 阶段 6：Skill 完全重写

#### 6.1 5 个 SKILL.md 重写原则

参考 Cowart `skills/cowart-image-gen/SKILL.md`，每个 SKILL.md 必须包含：

1. **frontmatter**：`compatibility` 改为 `opencode, zcode, mcp`（删 codex、claude-code）
2. **Preconditions**：服务必须先启动（`vibecanvas serve` 或 `vibecanvas mcp`），projectDir 已配
3. **Workflow**（编号步骤）：每步给
   - 调哪个 MCP 工具
   - 入参 JSON（含字段从上一步返回的哪个字段取）
   - 返回值的关键字段
   - 出错怎么办
4. **Constraints**：硬规则（revision CAS、不要全图重写、高成本异步轮询等）
5. **Notes**：边界情况、provider profile 细节、性能建议

#### 6.2 新增 `references/tool-cookbook.md`

每个 skill 目录下加 `references/tool-cookbook.md`，写完整参数级示例：

- `place_artifact` + `set_artifact_status` 收尾
- `inspect_artifact` / `list_artifacts` / `list_templates` / `apply_template` 各自示例

#### 6.3 补齐 7 个遗漏工具的章节

## 执行顺序建议

按风险递增：先做阶段 2（删 Web 反向，破坏性最低）→ 阶段 3（删 bridge 文件）→ 阶段 1（建 LLM 层，此时 runner 已经能 import 新抽象）→ 阶段 4-5-6-7。

实际我会按阶段 1 → 2 → 3 → 4 → 5 → 6 → 7 的顺序，因为阶段 1 完成后阶段 2-3 是机械删除，更安全。
